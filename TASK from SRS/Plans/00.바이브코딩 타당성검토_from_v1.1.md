# 바이브코딩 타당성 검토

**Date:** 2026-05-23  
**Context:** SRS v1.1 → v2.0 스택 마이그레이션 계획서 기반  
**대상 프로필:** 입문자 수준 SW 개발 배경지식을 갖춘 7년차 서비스 기획 직무 종사자  

---

## 검토 배경

SRS v1.1 기반의 "안심 CBT 역직구 플랫폼" MVP를 Next.js 풀스택(App Router + Prisma + Supabase + Vercel) 기술 스택으로 전환한 Migration Plan을 작성한 후, 해당 내용이 아래 두 가지 개발 방식에 적절한지를 개발 난이도 및 구현 가능성 관점에서 검토한다.

- **방식 A:** 완전 바이브코딩 기반의 MVP 개발 작업
- **방식 B:** 엔지니어링 지식을 투입하며 AI 생성 결과물을 개선할 수 있는 MVP 개발 작업

---

## 방식 A: 완전 바이브코딩 기반 MVP 개발

"프롬프트로 지시 → AI가 코드 생성 → 결과물을 그대로 사용"하는 방식을 전제한다.

### 가능한 영역 (전체의 약 40~50%)

F5(정품 배지 표시), F6(건기식 설명 카드), F7(빠른 재주문)은 바이브코딩으로 충분하다. DB에서 데이터 읽어서 화면에 뿌리는 CRUD 패턴이고, shadcn/ui 컴포넌트 조합으로 UI가 나오며, Prisma 쿼리도 단순하다. Admin Dashboard의 테이블·폼 CRUD도 같은 범주이다.

### 어려운 영역 (나머지 50~60%)

**F1 총비용 계산 — 난이도 높음.** 단순히 "가격 보여줘"가 아니라, 환율 API 호출 → 캐시 히트/미스 분기 → TTL 초과 시 결제 차단 → disclaimer 분리 표시 → 비지원국 분기 처리가 하나의 요청 안에서 다 일어나야 한다. 바이브코딩으로 "환율 API 연동해서 총비용 계산해줘"라고 하면 정상 경로는 만들어지지만, 캐시 폴백·TTL 만료·API 타임아웃 같은 **실패 경로 6개**를 AI가 자발적으로 구현할 가능성이 낮다. 그런데 이 실패 경로가 PRD의 핵심 가치("숨겨진 비용 없는 안심")를 지키는 장치이다.

**F3 주소 검증 — 난이도 중상.** 실시간 디바운스 + 클라이언트 1차 검증 + 서버 2차 검증 + 외부 API 폴백 + 국가별 Regex 분기를 한 흐름으로 엮어야 한다. "주소 검증 폼 만들어줘"로는 국가별 분기와 폴백 체인이 빠진다.

**F4 알림 재시도 큐 — 난이도 중상.** Vercel Cron + DB 폴링 + 재시도 간격 계산 + 에스컬레이션 분기가 필요하다. 바이브코딩으로 Cron Job 자체는 만들 수 있지만, "retry_count별로 next_retry_at을 다르게 계산하고, 3회 실패 시 CS 에스컬레이션"이라는 상태 머신을 정확하게 만들려면 여러 번의 수정 사이클이 필요하다.

**Vercel KV 캐시 레이어 — 난이도 중.** 환율·통관 판정의 캐시 읽기/쓰기/만료 처리를 서비스 로직 곳곳에 끼워 넣어야 하는데, AI에게 "캐시 추가해줘"라고 하면 한 곳에만 넣거나 TTL 로직을 빠뜨리는 경우가 잦다.

### 종합 판정 — 방식 A

**현 SRS/Plan 수준은 바이브코딩만으로는 과도하다.** 이유는 SRS가 폴백·장애 내성·상태 분기를 매우 상세하게 정의하고 있기 때문이다. 이건 7년차 기획자의 "좋은 기획력"이 반영된 결과인데, 역설적으로 그 상세한 요구사항을 바이브코딩으로 정확히 구현하기가 어렵다.

바이브코딩으로 가려면 SRS 자체를 **대폭 간소화**해야 한다. 구체적으로는 모든 폴백 시나리오(AC1-F1, AC1-F2, AC2-F1, AC2-F2, AC3-F1, AC3-F2, AC4-F1, AC4-F2, AC5-F1, AC5-F2)를 MVP에서 제외하고 "정상 경로만 동작하는 버전"으로 스코프를 줄여야 한다. 캐시 레이어(Vercel KV)도 초기에는 빼고, 환율 API가 실패하면 그냥 에러를 보여주는 식으로 단순화해야 한다. 그러면 바이브코딩으로도 2~3주 안에 동작하는 프로토타입이 나올 수 있다. 단, 이 경우 PRD가 말하는 "불확실성 제거 엔진"의 핵심 가치(장애 상황에서도 안심)가 약해진다.

## 방식 B: 엔지니어링 지식 투입 + AI 결과물 개선

"AI가 코드 초안 생성 → 본인이 구조를 이해하고 수정·보완"하는 방식이다.

### 현 SRS/Plan과의 적합성 — 상당히 좋음

이 방식이라면 현 SRS 수준이 적절하다. 이유가 몇 가지 있다.

첫째, 기술 스택 선택이 이 방식에 최적화되어 있다. Next.js + Prisma + shadcn/ui는 AI 코드 생성의 품질이 가장 높은 조합이고, 생성된 코드를 읽고 수정하기에도 진입장벽이 낮다. Server Action 패턴은 "함수 하나가 폼 제출을 처리한다"는 멘탈 모델이 직관적이어서 서비스 기획자가 이해하기 좋다.

둘째, SRS의 상세한 AC/폴백 정의가 **AI에게 주는 프롬프트 재료**로 직접 쓸 수 있다. 예를 들어 "REQ-FUNC-104: 환율 API가 3초 초과하면 Vercel KV에서 캐시값을 읽고, TTL이 2시간 초과면 결제를 차단하고 메시지를 표시해"라는 AC를 그대로 프롬프트에 넣으면 AI가 상당히 정확한 초안을 만든다. 기획자가 AC를 잘 쓸 수 있다는 게 이 방식에서 가장 큰 강점이다.

### 그래도 도전적인 부분

**Prisma 스키마 설계 + 마이그레이션** — 12개 테이블의 관계(1:N, N:M), 인덱스 전략, JSONB 필드 설계를 한 번에 맞추기 어렵다. AI가 초안을 만들지만 "이 쿼리가 왜 느린지"를 판단하려면 DB 기초 지식이 필요하다. 다만 입문자 수준이라도 Prisma의 선언적 스키마는 SQL보다 훨씬 읽기 쉬워서 학습 곡선이 완만하다.

**Vercel Cron + 상태 머신** — 재시도 큐의 상태 전이(PENDING → SENT/FAILED → ESCALATED)를 정확하게 구현하려면 DB 트랜잭션과 경쟁 조건(Cron이 동시에 두 번 실행되면?)을 이해해야 한다. AI가 만든 코드를 "이게 동시 실행되면 어떻게 되지?"라는 관점에서 검토할 수 있어야 한다.

**외부 API 연동 디버깅** — Stripe, Open Exchange Rates, 물류사 API의 에러 응답을 해석하고 대응하는 건 AI보다 본인이 직접 해야 할 영역이다. API 문서를 읽고 "이 에러 코드가 뭔지" 찾는 능력이 필요하다.

### 종합 판정 — 방식 B

**현 SRS/Plan 수준이 적절하며, 이 방식이 권장된다.** 다만 몇 가지 조정을 제안한다.

**구현 순서 재배치가 필요하다.** 현 Migration Plan §8은 SRS 문서 수정을 먼저 하고 구현에 들어가는 순서인데, 방식 B에서는 "쉬운 기능부터 구현하며 스택에 익숙해지기 → 어려운 기능 순차 도전"이 효과적이다. 구체적으로는 F5(배지 표시) → F6(설명 카드) → F7(재주문) → F3(주소 검증) → F2(통관 판정) → F1(총비용) → F4(알림·추적) 순서가 학습 곡선에 맞다.

**폴백 시나리오는 2차 스프린트로 분리하는 게 현실적이다.** 1차에서 정상 경로를 완성하고, 2차에서 캐시 폴백·API 장애 처리·재시도 큐를 추가하면 각 단계의 복잡도가 관리 가능해진다. SRS의 AC-F1, AC-F2류 폴백 AC들을 "Phase 1.5"로 명시적으로 분리하면 된다.

## 결론

| 기준 | 방식 A (완전 바이브코딩) | 방식 B (AI + 엔지니어링 개선) |
|---|---|---|
| 현 SRS/Plan 난이도 적합성 | ❌ 과도함 — 간소화 필요 | ✅ 적절함 |
| MVP 핵심 가치 보존 가능성 | ⚠️ 폴백 제거 시 가치 약화 | ✅ 전체 보존 가능 |
| 7년차 기획자 실현 가능성 | ⚠️ 정상 경로만 가능 | ✅ 학습 투자 시 전체 가능 |
| 권장 조치 | SRS 스코프 축소 (폴백·캐시 제거) | 구현 순서 재배치 + 폴백 2차 분리 |

방식 B를 택한다면 현 문서를 그대로 활용하되 구현 순서만 조정하면 된다. 방식 A를 택한다면 SRS를 "정상 경로 전용 Lite 버전"으로 별도 추출하는 작업이 먼저 필요하다.
