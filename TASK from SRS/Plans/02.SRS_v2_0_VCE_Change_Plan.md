## 1. 개요

### 1.1 목적

본 문서는 SRS v2.0 Vibe Coding Edition(v1.0)에 아래 두 가지 결정사항을 반영하여 v1.1로 갱신하기 위한 구체적 작업 계획과, 변경 후 MVP 핵심 사용자 경험이 훼손되지 않는지에 대한 검증 결과를 기록한다.

### 1.2 적용 결정사항

| # | 결정 | 근거 |
|---|---|---|
| D-1 | **C-TEC-005/006(Vercel AI SDK + Gemini API)을 제약으로 등록하되, MVP에서는 LLM 기능 미적용 (옵션 A)** | F1~F7 기능 요구사항 중 LLM에 직접 의존하는 것이 없음. 바이브코딩 MVP에서 LLM 연동까지 추가하면 복잡도가 불필요하게 상승. 스택만 준비하고 기능 적용은 Phase 2로 배치 |
| D-2 | **F3 AddressFormat 시드 데이터(5개국) 추가** | 바이브코딩 시 AI가 국가별 우편번호·전화번호 정규식을 할루시네이션할 위험이 있음. 검증된 시드 데이터를 SRS에 명시하면 AI 코드 생성 품질이 직접적으로 향상됨 |

### 3.1 C-1: §1.1 Purpose 간소화 원칙

**Before (v1.0):**
```
간소화 원칙:
- 정상 경로(Happy Path)만 명세한다.

```
간소화 원칙:
- 정상 경로(Happy Path)만 명세한다.

### 3.4 C-5: §5.3 시드 데이터 (신설 섹션)

**신규 섹션 전문:**

> ### 5.3 시드 데이터: AddressFormat (5개국)
>
> 아래 데이터는 Prisma seed 스크립트(`prisma/seed.ts`)로 초기 투입한다.
> 바이브코딩 시 이 테이블을 그대로 시드 파일에 포함한다.

| countryCode | 국가 | postalRegex | phoneRegex | requiredFields |
|---|---|---|---|---|
| US | 미국 | `^\d{5}(-\d{4})?$` | `^\+1\d{10}$` | `["addressLine1","city","state","postalCode"]` |
| JP | 일본 | `^\d{3}-?\d{4}$` | `^\+81\d{9,10}$` | `["addressLine1","city","postalCode"]` |
| AU | 호주 | `^\d{4}$` | `^\+61\d{9}$` | `["addressLine1","city","state","postalCode"]` |
| SG | 싱가포르 | `^\d{6}$` | `^\+65\d{8}$` | `["addressLine1","postalCode"]` |
| CA | 캐나다 | `^[A-Za-z]\d[A-Za-z] ?\d[A-Za-z]\d$` | `^\+1\d{10}$` | `["addressLine1","city","state","postalCode"]` |

> 위 5개국은 PRD v0.4의 "대표 5개국" 예시이며, 실제 대상국 확정 시 교체한다.

## 4. MVP 핵심 사용자 경험 보존 검증

### 4.1 검증 원칙

PRD v0.4가 정의한 핵심 가치는 **"5대 구매 후 불확실성의 결제 전 선제 제거"**이다. 이번 변경(D-1 LLM Phase 2 배치, D-2 시드 데이터 추가)이 F1~F7 각 기능의 가치 전달을 훼손하는지를 검증한다.

검증 축:
1. **기능적 동등성** — 변경 전후로 사용자가 받는 기능이 동일한가?
2. **데이터 정확성** — 시드 데이터 추가가 기존 데이터 모델이나 로직에 부정적 영향을 주는가?
3. **구현 복잡도** — 변경이 바이브코딩 구현 난이도를 올리는가?

### 4.2 D-1 검증: LLM 미적용이 사용자 경험에 미치는 영향

| 기능 | LLM 없이 제공되는 경험 | LLM이 있으면 추가되는 경험 (Phase 2) | 훼손 여부 |
|---|---|---|---|
| **F1 총비용** | DB 룰 테이블 + 환율 API → TLC 항목별 표시 | 변화 없음 (LLM 무관) | ✅ 훼손 없음 |
| **F2 통관** | DB 룰 → ✅/⚠️/❌ 배지 + 조건 안내 | 변화 없음 (LLM 무관) | ✅ 훼손 없음 |
| **F3 주소** | Regex 기반 실시간 검증 | 변화 없음 (LLM 무관) | ✅ 훼손 없음 |
| **F4 배송** | 표준 상태 매핑 + 예외 가이드 텍스트 (정적) | LLM으로 상황별 맞춤 가이드 자동 생성 가능 | ✅ 훼손 없음 — 정적 가이드 텍스트로 PRD AC4-3 충족 |
| **F5 정품** | 인증 배지 + 유통기한 표시 | 변화 없음 (LLM 무관) | ✅ 훼손 없음 |
| **F6 건기식** | 운영팀이 수동 입력한 구조화 데이터 표시 | LLM으로 한국어→영어 자동 번역 가능 | ✅ 훼손 없음 — 수동 입력으로 PRD REQ-F6-01~03 충족. 번역은 편의 개선이지 핵심 가치가 아님 |
| **F7 재주문** | 주문 복제 + 상태 변경 알림 | 변화 없음 (LLM 무관) | ✅ 훼손 없음 |

**D-1 결론:** LLM이 직접적으로 가치를 전달하는 기능은 F1~F7에 없다. LLM은 F4(맞춤 가이드)와 F6(자동 번역)에서 "편의 개선"을 제공할 수 있지만, 핵심 가치(불확실성 제거)는 DB 룰 테이블 + 정적 데이터로 충분히 전달된다. **MVP 사용자 경험 훼손 없음.**

### 4.3 D-2 검증: AddressFormat 시드 데이터 추가가 미치는 영향

| 검증 항목 | 결과 | 근거 |
|---|---|---|
| 기능적 동등성 | ✅ 보존 | 시드 데이터는 AddressFormat 테이블에 초기 데이터를 넣는 것일 뿐, F3 기능 요구사항(REQ-F3-01~03)에 변화 없음 |
| 데이터 정확성 | ✅ 개선 | AI가 임의로 생성하던 Regex 패턴 대신 검증된 패턴을 사용하므로, 주소 검증의 정확도가 향상됨 |
| 구현 복잡도 | ✅ 감소 | 바이브코딩 시 "이 시드 데이터를 prisma/seed.ts에 넣어줘"라는 단순 프롬프트로 구현 가능. AI가 Regex를 직접 만들 필요가 없어져 할루시네이션 리스크 제거 |
| ERD 영향 | ✅ 없음 | AddressFormat 테이블은 이미 ERD에 존재. 스키마 변경 없이 데이터만 추가 |
| 다른 기능 영향 | ✅ 없음 | AddressFormat은 F3에서만 참조. F1, F2, F4~F7에 영향 없음 |

**D-2 결론:** 시드 데이터 추가는 F3의 구현 품질을 높이고, 다른 기능에 부정적 영향을 주지 않는다. **MVP 사용자 경험 훼손 없음. 오히려 주소 검증 정확도 개선으로 가치 강화.**

### 4.4 종합 판정

| 변경 | 영향받는 기능 | 가치 전달 훼손 | 가치 전달 강화 |
|---|---|---|---|
| D-1 LLM Phase 2 배치 | 없음 (F1~F7 무관) | ❌ 훼손 없음 | — |
| D-2 시드 데이터 추가 | F3 주소 검증 | ❌ 훼손 없음 | ✅ 검증 정확도 향상 |

> **최종 결론:** 두 변경사항 모두 MVP 핵심 사용자 경험("5대 불확실성의 결제 전 선제 제거")을 훼손하지 않는다. D-2는 오히려 F3의 품질을 개선한다. PRD v0.4의 북극성 KPI(거래 완료율 ≥ 90%)에 기여하는 F1~F7의 정상 경로는 변경 전후 100% 동일하게 유지된다.
