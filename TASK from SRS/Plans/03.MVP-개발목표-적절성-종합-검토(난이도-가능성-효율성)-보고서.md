## 1. 검토 범위 및 기준

### 1.1 검토 대상

SRS v2.0 Vibe Coding Edition v1.1이 정의한 MVP — F1~F7 기능 7개, 데이터 모델 9개 테이블, API 12개 엔드포인트, 외부 연동 3개(Stripe, Open Exchange Rates, 물류사 Tracking API)를 Next.js + Prisma + Supabase + Vercel 스택으로 바이브코딩 구현하는 것.

### 1.2 검토 축

| # | 검토 축 | 핵심 질문 |
|---|---|---|
| 1 | **개발 속도** | MVP 단계에서 과도한 시간 소요 없이 구현 가능한가? 기능 범위, 테이블 수, 외부 연동 수가 바이브코딩 1인 개발에 적절한가? |
| 2 | **외부 연동 목표와 기술 스택** | 외부 API 선택이 복잡성·오픈소스 여부·안정성 면에서 적절한가? 기술 스택 조합에 숨은 리스크는 없는가? |

### 2.4 개발 속도 판정

**✅ 적절함.** 25일(5주)은 1인 바이브코딩 MVP로서 합리적인 범위이다. 원본 SRS v1.1 기준이었다면 폴백 시나리오·캐시·재시도 큐까지 포함되어 8~12주가 소요되었을 것이다. 현재 SRS v2.0 VCE는 이를 정상 경로로 압축하여 시간을 약 50~60% 단축했다.

**추가 조정 필요 여부: 없음.** 기능 범위를 더 줄이면(예: F4 배송 추적 제거) 3~4주로 단축 가능하지만, PRD의 5대 불확실성 중 "배송 불확실성(P1)"을 포기하게 되어 가치 훼손이 발생한다. 현재 7개 기능이 MVP 최소 단위이다.

## 3. 검토 축 2: 외부 연동 목표와 기술 스택

### 3.1 외부 API별 적절성 평가

#### 3.1.1 EXT-01: Open Exchange Rates API (환율)

| 항목 | 평가 |
|---|---|
| 용도 | F1 총비용의 현지 통화 변환 |
| 무료 티어 | 1,000 req/월, 시간별 갱신, USD 기반만 |
| 유료 | $12/월~ (Developer), 모든 base currency |
| 오픈소스 대안 | exchangerate.host (무료, 일 1회 갱신), ECB 공개 데이터 |
| 복잡도 | 🟢 낮음 — REST GET 1개, JSON 응답 |
| 리스크 | 무료 티어 USD 기반 제한. KRW→USD→현지통화 2단계 변환 필요 |

**판정: ⚠️ 조건부 적절.**

무료 티어의 "USD base only" 제한이 문제이다. 한국 상품의 기본 가격이 KRW이므로, KRW→현지통화 직접 변환이 안 되고 KRW→USD→현지통화 2단계 계산이 필요하다. 이 자체는 코드로 해결 가능하지만, 비용 계산 오류율(REQ-NF-05 ≤ 1%)에 영향을 줄 수 있다.

**권장 조정:**
- MVP 초기: 무료 티어로 시작. 월 1,000건이면 Alpha~Closed Beta(60명)에서 1인당 약 16회 환율 조회 가능. 충분하다.
- Open Beta(500명): Developer 플랜($12/월) 업그레이드. 모든 base currency 지원되어 KRW 직접 변환 가능.
- SRS 변경 필요: **없음.** 현재 SRS는 API 선택을 강제하지 않고 "환율 API"로만 명시. 구현 시 프롬프트에서 구체 API를 지정하면 됨.

#### 3.1.2 EXT-02: 물류사 Tracking API (배송)

| 항목 | 평가 |
|---|---|
| 용도 | F4 배송 상태 폴링 + ETA |
| 비용 | 물류 파트너 계약에 따라 상이 (보통 무료 제공) |
| 오픈소스 대안 | 없음 (물류사 고유 API) |
| 복잡도 | 🟡 중간 — 물류사마다 응답 형식 상이, 표준화 매핑 필요 |
| 리스크 | ASM-04 "물류 파트너가 API를 제공함" 가정에 의존 |

**판정: ✅ 적절.**

물류사 API는 선택의 여지가 없다(파트너가 제공하는 것을 쓴다). SRS가 "표준 6단계 매핑"을 요구하는 것은 올바른 설계이다. 다만 MVP 초기에 물류 파트너가 확정되지 않았다면, **Mock API로 시작하는 것도 방법**이다.

**권장 조정:**
- SRS §1.5 ASM-04에 "물류 파트너 미확정 시 Mock API로 개발·테스트" 주석 추가 고려. 다만 필수는 아님.

#### 3.1.3 EXT-03: Stripe (결제)

| 항목 | 평가 |
|---|---|
| 용도 | F1 결제 처리, 다통화 지원 |
| 비용 | 건당 2.9% + $0.30 (표준), 국제 카드 추가 1% |
| 오픈소스 대안 | 없음 (PG는 오픈소스 불가) |
| 복잡도 | 🟢 낮음 — Stripe SDK가 매우 잘 문서화됨, AI 코드 생성 품질 최상급 |
| 리스크 | 5개국 커버리지 확인 필요 (ASM-05). 한국 발 Stripe Connect 지원 여부 |

**판정: ✅ 적절.**

Stripe는 CBT 결제에서 사실상 표준이다. AI 코드 생성 품질도 가장 높은 API 중 하나이다. PCI DSS를 Stripe에 위임하는 것도 올바른 판단이다.

**권장 조정: 없음.**

### 3.2 기술 스택 조합 적절성

| 스택 구성 | 적절성 | 근거 |

### 3.3 스택 조합 숨은 리스크

| 리스크 | 영향 | 심각도 | 완화 |
|---|---|---|---|
| **Prisma Cold Start on Vercel Serverless** | Prisma Client 초기화가 Serverless에서 느릴 수 있음 (500ms~1.5초) | 🟡 중간 | Prisma Accelerate(유료) 또는 connection pooling 설정. MVP 초기 트래픽에서는 체감 영향 미미 |
| **SQLite ↔ PostgreSQL JSON 쿼리 차이** | 로컬에서 통과한 JSON 필드 쿼리가 배포에서 다르게 동작 | 🟡 중간 | JSON 내부 필터링은 애플리케이션 레벨로 제한 (SRS §5 이미 반영) |
| **Vercel Serverless 함수 타임아웃** | Pro 플랜 60초 제한. 복잡한 TLC 계산 + 환율 API 호출이 겹치면 초과 가능 | 🟢 낮음 | MVP 규모(20~30 SKU)에서는 발생 가능성 극히 낮음 |
| **Supabase 무료 티어 프로젝트 일시정지** | 7일 비활동 시 자동 일시정지 | 🟡 중간 | Pro 플랜($25/월)으로 시작하면 해결. MVP 비용에 이미 포함 |

### 3.4 외부 연동·기술 스택 판정

**✅ 적절함. 추가 조정 필요 없음.**

외부 API 3개(환율·물류·결제)는 MVP 최소 연동 수이며, 각각의 선택이 합리적이다. 기술 스택 조합(Next.js + Prisma + Supabase + Vercel)은 바이브코딩에 가장 최적화된 조합으로, AI 코드 생성 품질·학습 곡선·운영 단순성 모두에서 우수하다. 숨은 리스크 4개가 있지만 MVP 규모에서 심각한 것은 없다.

### 5.2 최종 결론

> **SRS v2.0 Vibe Coding Edition v1.1은 개발 속도·외부 연동·운영 비용 3개 축 모두에서 MVP 개발 목표에 적절하며, 추가 조정이 필요하지 않다.**

구체적으로:

1. **개발 속도:** 정상 경로만 명세하고 폴백·캐시·재시도를 Phase 2로 배치한 것이 바이브코딩 1인 개발에 정확히 맞는 스코프 관리이다. 약 5주라는 타임라인은 "빠르지도 느리지도 않은" 현실적 범위이다.

2. **기술 스택:** Next.js + Prisma + Supabase + Vercel 조합은 2026년 기준 바이브코딩에서 AI 코드 생성 품질이 가장 높고, 학습 곡선이 가장 완만한 조합이다. 외부 API 3개(Stripe, 환율, 물류)는 MVP에 필수적인 것만 남긴 최소 구성이다.

3. **운영 비용:** 월 $46~$80의 인프라 비용은 MVP 단계에서 재정적 부담이 거의 없으며, GA까지 확장해도 $100 이내를 유지할 수 있다. PRD 원본의 AWS 구조($3,000/월) 대비 97% 절감이다.
