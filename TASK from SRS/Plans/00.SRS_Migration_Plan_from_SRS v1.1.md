## 1. 개요

### 1.1 목적

본 문서는 SRS v1.1에 정의된 "모듈러 모놀리스 + 별도 API Gateway + Redis + AWS" 아키텍처를 **Next.js App Router 기반 단일 풀스택 프레임워크 + Vercel + Supabase** 스택으로 전환하기 위한 구체적 변경 계획을 수립한다.

변경의 핵심 원칙은 다음과 같다.

- PRD v0.4에서 정의한 **5대 불확실성 제거 가치**와 **F1~F7 MVP 기능의 사용자 경험은 100% 보존**한다.
- 기술 스택만 변경하며, 비즈니스 요구사항(REQ-FUNC-*)은 변경하지 않는다.
- NFR 중 인프라·도구 의존적 항목만 재정의하고, SLO 수치 목표는 유지한다.

### 1.2 목표 기술 스택 (C-TEC-001 ~ C-TEC-007)

| ID | 제약 | 상세 |
|---|---|---|
| C-TEC-001 | 단일 풀스택 프레임워크 | 모든 서비스는 Next.js (App Router) 기반. 프론트엔드·백엔드 분리 없음 |
| C-TEC-002 | 서버 측 로직 구현 방식 | Server Actions 또는 Route Handlers 사용. 별도 백엔드 서버 없음 |
| C-TEC-003 | 데이터베이스 | Prisma ORM + SQLite(로컬 개발) / Supabase PostgreSQL(배포) |
| C-TEC-004 | UI·스타일링 | Tailwind CSS + shadcn/ui |
| C-TEC-005 | LLM 오케스트레이션 | Vercel AI SDK (Next.js 내부에서 직접 구현) |
| C-TEC-006 | LLM 모델 | Google Gemini API 기본. 환경 변수로 모델 교체 가능 |
| C-TEC-007 | 배포·인프라 | Vercel 플랫폼 단일화. Git Push 자동 배포 |

## 4. 핵심 기술 결정 사항 (변경 시 결정 필요)

### 4.1 Decision 1: 캐시 전략 — Redis 대체

**컨텍스트:** 현행 SRS는 Redis로 환율 캐시(TTL 1h), 통관 판정 캐시, 세션을 관리한다. Vercel+Supabase 스택에는 Redis가 기본 포함되지 않는다.

**영향받는 요구사항:**
- REQ-FUNC-104: 환율 API 타임아웃 시 캐시 폴백 (TTL ≤ 2h)
- REQ-FUNC-205: Compliance Engine 장애 시 캐시 폴백
- REQ-NF-013: 외부 API 장애 내성 (환율·세금 캐시 TTL ≤ 2h)

**선택지:**

| 옵션 | 설명 | 장점 | 단점 |
|---|---|---|---|
| **A. Vercel KV** | Vercel 네이티브 Redis 호환 KV 스토어 | Redis와 동일 API, TTL 지원, 기존 로직 거의 그대로 이식 | 추가 비용 ($1/100K req), 별도 서비스 의존 |
| B. Supabase DB 캐시 테이블 | `exchange_rate_cache`, `compliance_cache` 테이블 + expires_at 칼럼 | 추가 서비스 없음, Prisma로 통합 관리 | DB 부하 증가, TTL 자동 만료 없음 (Cron으로 정리) |
| C. Next.js unstable_cache | Next.js 빌트인 데이터 캐시 (revalidate) | 추가 서비스 없음 | Serverless 인스턴스 간 비공유, 일관성 문제 |

**권장:** **옵션 A (Vercel KV)** — REQ-FUNC-104의 "캐시 폴백 성공률 ≥ 99%" SLO를 안정적으로 충족하려면 인스턴스 간 공유 캐시가 필수이며, Vercel KV가 기존 Redis 로직을 최소 변경으로 이식 가능하다. 초기 트래픽에서 비용 부담은 미미하다.

**대안 트리거:** MVP 초기 트래픽이 매우 적어 비용 최적화가 최우선이면 옵션 B로 시작 후 A로 전환한다.

### 4.2 Decision 2: 비동기 작업 처리 — 재시도 큐·폴링

**컨텍스트:** Vercel Serverless Functions는 실행 시간 제한(Hobby 10초, Pro 60초)이 있어 장시간 백그라운드 작업(알림 재시도 5분·15분·60분 간격, 물류 폴링 1시간 간격, 48시간 무응답 체크)을 직접 처리할 수 없다.

**영향받는 요구사항:**
- REQ-FUNC-404: 물류 Tracking API 폴링 + 48시간 CS 티켓 자동 생성
- REQ-FUNC-405: 알림 재시도 큐 (3회, 5분·15분·60분)
- REQ-FUNC-406: 물류사 폴링 (≥ 1시간 간격)
- REQ-NF-033~035: Health Check (1분 주기)

**설계:**

```
┌──────────────────────────────────────────────────────────┐
│  Vercel Cron Jobs (vercel.json)                          │
│                                                          │
│  "*/1 * * * *"  → /api/cron/retry-notifications          │
│                    DB에서 pending 알림 조회 → 재시도      │
│                    retry_count별 재시도 간격 충족 시 실행  │
│                                                          │
│  "0 * * * *"    → /api/cron/tracking-poll                │
│                    활성 주문의 물류 Tracking API 폴링      │
│                    48시간 무응답 → CS 티켓 자동 생성       │
│                                                          │
│  "*/1 * * * *"  → /api/cron/health-check                 │
│                    외부 API(환율·세금·주소) 상태 체크      │
│                    연속 3회 실패 → Slack Webhook 알림      │
└──────────────────────────────────────────────────────────┘
```

**재시도 큐 상세 설계:**

| 단계 | 기존 SRS (Redis 큐) | 변경 후 (DB 폴링) |
|---|---|---|
| 알림 발송 실패 | Redis 큐에 enqueue | Notifications 테이블에 status=PENDING, retry_count++ 기록 |
| 재시도 간격 판별 | Redis TTL/delayed queue | `next_retry_at` 칼럼 계산 (retry_count=1→+5분, 2→+15분, 3→+60분) |
| 재시도 실행 | 별도 Worker 프로세스 | Vercel Cron (1분 간격) → `WHERE status='PENDING' AND next_retry_at <= NOW()` 조회 후 발송 |
| 에스컬레이션 | Worker에서 CS 대기열로 전달 | retry_count=3 AND status=FAILED → CS 에스컬레이션 로직 실행 |

**REQ-FUNC-405 SLO 충족 검증:**
- "에스컬레이션 지연 ≤ 70분" → 5+15+60=80분이 이론적 최대이지만, Cron이 1분 간격이므로 실제 최대 약 81분. 현행 SRS의 70분 목표와 1분 이내 오차로 충족 가능. 다만, Cron 실행 지연(Vercel은 보장하지 않음)을 고려해 SLO를 **≤ 85분**으로 완화하거나, 간격을 4분·14분·59분으로 조정하는 것을 검토한다.

### 4.3 Decision 3: 모니터링 도구 재매핑

| 현행 도구 | 대체 도구 | 커버리지 |
|---|---|---|
| CloudWatch / Datadog APM | **Vercel Analytics + Vercel Speed Insights** | 응답 시간 p95, 페이지 로드 |
| Application 로그 (CloudWatch) | **Vercel Logs + Sentry** | 에러 추적, 슬로우 쿼리, 비즈니스 로그 |
| PagerDuty | **Sentry Alerts + Slack Webhook** | 임계치 초과 알림 (비용 계산 오류 > 1%, 결제 실패 > 2%) |

| 수동 검수 대시보드 | **Admin Dashboard 내 모니터링 페이지** (/admin/monitoring) | 통관 판정 실패, 룰 미등록 요청 |

### 4.4 Decision 4: DB 타입 호환성 — JSONB 처리

**문제:** SQLite는 JSONB 타입을 지원하지 않는다. 현행 SRS ERD에서 JSONB를 사용하는 필드가 12개 이상이다.

| 테이블 | JSONB 필드 | Prisma 처리 방식 |
|---|---|---|
| Products | name_i18n, ingredients, price_rules | `Json` 타입 (Prisma가 SQLite에서는 String으로, PostgreSQL에서는 JSONB로 자동 매핑) |
| Orders | items, eta_info, exceptions | 동일 |
| Customers | addresses | 동일 |
| ComplianceRules | required_docs | 동일 |

**Prisma의 `Json` 타입이 양쪽 DB에서 동작하므로 코드 변경 없이 호환 가능.** 단, SQLite에서는 JSON 내부 필드 기반 쿼리 (예: `WHERE name_i18n->>'en' LIKE '%serum%'`)가 지원되지 않으므로, 이런 쿼리가 필요한 경우 애플리케이션 레벨에서 필터링한다.

### 4.5 Decision 5: 인증·세션 관리

**현행:** SRS §6.7에서 API Gateway의 인증 역할을 명시하지만 구체 구현을 정의하지 않는다.

**변경:**
- Next.js Middleware에서 인증 토큰 검증
- 세션 관리: NextAuth.js (Auth.js) 또는 Supabase Auth 사용
- Admin 접근 제어 (REQ-NF-018 RBAC): Middleware + DB 역할 칼럼으로 구현

## 6. MVP 기능별 사용자 경험 보존 검증

### 6.1 검증 원칙

PRD v0.4가 정의한 핵심 가치는 **"5대 구매 후 불확실성의 결제 전 선제 제거"**이다. 스택 변경이 이 가치 전달을 훼손하는지를 기능(F1~F7)별로 검증한다.

검증 기준:
1. **기능적 동등성** — 사용자가 동일한 입력에 동일한 출력을 받는가?
2. **응답 속도 SLO** — PRD/SRS에서 정의한 임계치를 충족하는가?
3. **폴백·장애 내성** — 외부 API 장애 시 동일한 폴백 경험을 제공하는가?
4. **사용자 여정 단절** — 여정 흐름(PDP→장바구니→결제→추적)에 끊김이 없는가?

### 6.2 F1: 결제 전 총비용 확정 (Clarity Job)
