## 1. 기술 스택의 명확성

SRS v2.0에서 가장 근본적인 변화는 **기술 스택의 전면 전환 및 구체적 명시**이다. v1.1은 기술 스택을 거의 정의하지 않은 반면, v2.0은 모든 레이어를 명확히 지정했다.

| # | 변경 항목 | SRS v1.1 (원본) | SRS v2.0 (VibeCoding Edition) | 변경 이유 / 평가 |
|---|---|---|---|---|
| 1 | **프론트엔드 프레임워크** | "Web Application (SPA)" — React 수준의 언급만 존재. 구체적 프레임워크 미지정 | **Next.js (App Router)** — SSR+CSR 하이브리드 명시 (CON-01) | 풀스택 단일 앱 구조를 통해 프론트엔드·백엔드 분리 금지를 제약으로 못 박아 바이브코딩 복잡도를 낮춤. 명확성 ↑ |
| 2 | **백엔드 구조** | "Modular Monolith" + API Gateway + 8개 서비스 모듈 (Product, Order, Pricing, Compliance, Fulfillment, Address, Notification, Admin) | **Server Actions + Route Handlers** (별도 백엔드 서버 금지, CON-02). `lib/` 하위 모듈로 로직 분리 | 모놀리스 내 도메인 서비스 분리 대신 Next.js 내장 서버 기능을 활용. 별도 API Gateway 제거. 명확성 ↑↑ |
| 3 | **데이터베이스** | PostgreSQL (AWS RDS 추정), Redis Cache (환율·통관·세션) | **Prisma ORM + SQLite(로컬) / Supabase PostgreSQL(배포)** (CON-03). 캐시 레이어 제거 | ORM 도구와 로컬/배포 DB를 명시적으로 분리 지정. Redis 제거로 인프라 단순화. 명확성 ↑↑ |
| 4 | **UI 프레임워크** | 미지정 (React만 언급) | **Tailwind CSS + shadcn/ui** (CON-04) | 컴포넌트 라이브러리까지 확정하여 UI 구현 시 의사결정 비용 제거. 명확성 ↑↑ |
| 5 | **배포 인프라** | AWS (CloudWatch, Datadog, PagerDuty, Slack 등) — 구체적 서비스 조합 미확정 | **Vercel** — Git Push 자동 배포 (CON-05) | 서버리스 배포로 인프라 관리 부담 제거. Vercel Edge Network 기반으로 CDN·SSL 자동 제공. 명확성 ↑↑ |
| 6 | **결제 시스템** | Stripe (PCI DSS 위임) — 동일 | Stripe (PCI DSS 위임) — **동일** (CON-08, CON-09) | 변경 없음. 양쪽 모두 Stripe 사용으로 일관 |
| 7 | **인증/권한** | "Admin Dashboard RBAC (역할 기반 접근 제어)" — 구현 도구 미지정 | **NextAuth.js 또는 Supabase Auth** 명시 (REQ-NF-08) | 인증 솔루션 후보를 구체적으로 2개로 좁힘. 명확성 ↑ |
| 8 | **모니터링** | CloudWatch/Datadog APM + PagerDuty/Slack 알림 (10개 Health Check 포함) | **Vercel 기본 제공 범위** + Sentry Free (5K 이벤트/월) | 10개의 상세 모니터링 요구 제거 → Vercel 기본 SLA 의존. 명확성은 유지하되 범위 축소 |
| 9 | **캐시 레이어** | Redis Cache (환율 TTL 1h, 통관 판정 캐시, 세션) | **없음** — 캐시 레이어 전량 Phase 2 이동 | MVP 아키텍처 단순화. 매 요청 시 DB/API 직접 조회 |
| 10 | **외부 API 연동 범위** | 5개 (환율 API, Google/Loqate 주소검증, 물류사 Tracking, Stripe, 알림 서비스) | **3개** (환율 API, 물류사 Tracking, Stripe만) | Google/Loqate, 이메일/푸시 알림 서비스 → Phase 2 이동. 연동 복잡도 40% 감소 |
| 11 | **LLM/AI 통합** | 미정의 (언급 없음) | **Vercel AI SDK + Google Gemini API** 스택 명시 (CON-10, CON-11). 단, MVP에서는 기능 미적용, Phase 2에서 활성화 | v1.1에 없던 AI 스택을 제약으로 사전 정의. 환경변수(`GOOGLE_GENERATIVE_AI_API_KEY`)까지 준비. 명확성 ↑↑ (신규) |
| 12 | **데이터 모델링 도구** | ERD + Java 스타일 Class Diagram (메서드 포함) | **Prisma Schema** — Class Diagram 제거 | Prisma 스키마 자체가 Single Source of Truth 역할. 별도 클래스 다이어그램 불필요 |
| 13 | **비용 산정** | "월 인프라 비용 ≤ $3,000 (초기 AWS)" — 상세 내역 없음 | **$46~$58/월** — Vercel $20 + Supabase $25 + 도메인 $1 + 기타. 서비스별 상세 산정표 포함 | 97% 비용 절감. 서비스별 플랜·포함 사항까지 명시. 명확성 ↑↑↑ |
| 14 | **Cron Job** | 3개 (tracking 폴링, retry 큐, health check) | **1개** (tracking 폴링만 — Vercel Cron) | retry 큐·health check Phase 2 이동. 비동기 백그라운드 작업 최소화 원칙 적용 |
| 15 | **환경변수** | 미정의 | **8개 환경변수** 목록 명시 (DATABASE_URL, STRIPE_SECRET_KEY 등 + Phase 2 준비용 GOOGLE_GENERATIVE_AI_API_KEY) | 바이브코딩 시 AI가 즉시 참조할 수 있도록 환경변수 목록화. 명확성 ↑↑ (신규) |

> **핵심 판단:** v2.0은 기술 스택에 대해 v1.1보다 **훨씬 높은 명확성**을 제공한다. v1.1이 "무엇(What)"을 정의했다면, v2.0은 "무엇을 어떤 도구로 어떻게(How)"까지 정의한다. 특히 비용 산정의 구체화($3,000 → $46~$58)와 LLM 스택 사전 정의는 주목할 만하다.

## 2. MVP 목표 및 가치전달 조정 내용

MVP 핵심 가치(5대 구매 후 불확실성 제거)는 유지하되, **정상 경로(Happy Path) 중심으로 스코프를 축소**하여 구현 가능한 수준으로 재설정했다.

| # | 변경 항목 | SRS v1.1 (원본) | SRS v2.0 (VibeCoding Edition) | 가치전달 영향 |
|---|---|---|---|---|
| 1 | **기능 요구사항 수** | 42개 (REQ-FUNC-101~1001) — 정상/장애/폴백 경로 모두 포함 | **25개** (REQ-F1-01~F7-02) — 정상 경로만 유지 | 40% 감소. **핵심 가치는 보존**되나 장애 내성 시나리오 전량 제거 |
| 2 | **NFR 수** | 45개 (REQ-NF-001~045) — 성능·신뢰성·보안·확장성·모니터링·KPI 전체 | **10개** (REQ-NF-01~10) | 78% 감소. 북극성 KPI(거래완료율 90%) 등 추적 지표 전량 제거 |
| 3 | **F1 총비용 — 폴백 처리** | 환율 API 타임아웃 시 캐시 폴백(TTL ≤ 2h), 캐시 만료 시 결제 차단, 비지원국 안내 (REQ-FUNC-104, 105) | **제거** — Phase 2 이동 | ⚠️ 환율 API 장애 시 결제 불가능. 초기 트래픽에서 체감 영향 미미하다는 가정 |
| 4 | **F2 통관 — 룰 미등록 처리** | 룰 미등록 시 ⚠️ + CS 링크 + 운영팀 자동 티켓 (REQ-FUNC-204) | **제거** — Phase 2 이동 | ⚠️ 미등록 국가/카테고리 접근 시 처리 미정의. 5개국·20~30 SKU 범위에서 미등록 확률 낮다는 가정 |
| 5 | **F2 통관 — 장애 내성** | Compliance Engine 장애 시 캐시 폴백 + 경고 표시 (REQ-FUNC-205) | **제거** — Phase 2 이동 | ⚠️ 서비스 장애 시 통관 판정 불가. MVP 안정성 리스크 |
| 6 | **F3 주소 — 외부 API 검증** | Regex 1차 + Google/Loqate 2차 검증. 장애 시 Regex 폴백 (REQ-FUNC-305) | **자체 Regex만** — 외부 API Phase 2 이동 | 검증 정확도 하락 가능. 그러나 시드 데이터로 5개국 Regex 패턴을 명시하여 보완 |
| 7 | **F3 주소 — libphonenumber** | libphonenumber 라이브러리 사용 명시 (REQ-FUNC-306) | **제거** — 자체 Regex로 대체 | 전화번호 검증 정밀도 하락. 국가별 phoneRegex 시드 데이터로 일부 보완 |
| 8 | **F4 배송 — 알림 발송** | 통관보류 10분 이내 알림 + 예외 해소 알림 + 재시도 큐(3회) + 에스컬레이션 (REQ-FUNC-401, 403, 405) | **제거** — 상태 페이지에서 수동 확인만 가능 | ⚠️ 능동적 알림 없이 고객이 직접 조회해야 함. **가치전달 약화** |
| 9 | **F5 정품 — 불일치 감지** | 인증 DB-카탈로그 불일치 감지 + 자동 티켓 생성 (REQ-FUNC-505) | **제거** — Phase 2 이동 | 인증 데이터 정합성 자동 관리 불가. 운영팀 수동 관리 의존 |
| 10 | **F5 정품 — 유통기한 미등록** | 미등록 시 플레이스홀더 + 운영팀 자동 알림 (REQ-FUNC-504) | 미등록 시 **플레이스홀더만 표시** — 자동 알림 제거 | 운영 효율 약간 하락. 플레이스홀더 표시로 핵심 가치는 유지 |
| 11 | **성능 목표** | TLC API p95 ≤ 1,000ms, Compliance p95 ≤ 500ms, 주소검증 p95 ≤ 300ms, 페이지 로드 p95 ≤ 2,000ms | 페이지 로드 ≤ 3초, API ≤ 2초, DB ≤ 500ms | 성능 기준 완화. Vercel Serverless 특성(Cold Start) 고려한 현실적 조정 |
| 12 | **신뢰성 목표** | 월 가용성 ≥ 99.5%, 비용 계산 오류율 ≤ 0.5%, 통관 정확도 ≥ 95% | 월 가용성 ≥ 99%, 비용 계산 오류율 ≤ 1% | 기준 완화. 통관 정확도 목표 제거 |
| 13 | **북극성 KPI** | 거래 완료율 Beta ≥ 82% → GA ≥ 90%, 전환율 ≥ 15%, NPS ≥ 40 등 8개 KPI 상세 정의 | **전량 제거** — NFR에 KPI 지표 없음 | ⚠️ 측정 기준 부재. 별도 KPI 문서에서 관리한다는 전제 필요 |
| 14 | **비용 목표** | 월 인프라 ≤ $3,000 + 건당 CS ≤ $2 | 월 인프라 **≤ $100** (실제 산정 $46~$58) | 30배 비용 절감 목표. CS 비용 목표는 제거 |
| 15 | **7대 기능 핵심 가치 유지 여부** | F1~F7 전체 구현 (정상+장애 경로) | F1~F7 전체 구현 (**정상 경로만**) | ✅ 7대 기능 모두 유지. 가치전달의 "폭"은 보존, "깊이"(장애 대응)만 축소 |

> **핵심 리스크:** F4 배송 추적에서 알림 시스템 제거는 가치전달에 가장 큰 영향을 미친다. 고객이 통관 보류·배송 지연 등 예외 상황을 **능동적으로 인지할 수단이 없어**, 직접 주문 상태 페이지를 방문해야 한다. MVP 초기 소수 고객에게는 수용 가능하나 Open Beta 이전에 반드시 Phase 2 알림 기능이 필요하다.

## 3. 기타 차이점

문서 구조, 데이터 모델, 부가 산출물 등 위 두 관점에 포함되지 않는 변경 사항을 정리한다.

| # | 변경 항목 | SRS v1.1 (원본) | SRS v2.0 (VibeCoding Edition) | 비고 |
|---|---|---|---|---|
| 1 | **문서 표준 적용 수준** | ISO/IEC/IEEE 29148:2018 전체 준수 | ISO/IEC/IEEE 29148:2018 **"간소화 적용"** | 바이브코딩 특성상 전체 표준 준수는 과도하다고 판단 |
| 2 | **문서 상태** | Draft | **Final Draft — 바이브코딩 MVP 구현 착수 가능** | 즉시 구현 가능 상태로 격상 |
| 3 | **데이터 모델 테이블 수** | 11개 (Brands, Products, Customers, Orders, OrderItems, ComplianceRules, PricingRules, AddressFormats, ShippingAddresses, TrackingEvents, Notifications, AuditLogs) | **9개** — Notifications, AuditLogs 제거 | 재시도 큐(Notifications)와 변경 이력(AuditLogs) Phase 2 이동 |
| 4 | **데이터 모델 타입 시스템** | PostgreSQL 네이티브 타입 (UUID, DECIMAL, JSONB, TIMESTAMP 등) | **Prisma 타입** (String, Float, Json, DateTime 등) | Prisma ORM 전환에 따른 자연스러운 변경 |

| 8 | **Customers 테이블** | `addresses` (JSONB) 필드 포함 | `addresses` 필드 **제거** | ShippingAddress 테이블로 정규화한 상태에서 JSONB 중복 제거 |
